# Calendar Recurring Events: Implementation Guide

This document explains how Mailspring displays, stores, and edits recurring calendar events, including exception handling. It is intended for developers working on the `main-calendar` package or the `Mailspring-Sync` C++ engine.

---

## 1. Core Concepts

### Recurring Event Terminology (RFC 5545)

| Term | Meaning |
|---|---|
| **Master event** | A VEVENT with an RRULE (and/or RDATE) that defines the recurrence pattern. Has an empty `recurrenceId`. |
| **Occurrence** | A single instance of the series, generated by expanding the RRULE. Not stored as a separate database row unless it is an exception. |
| **Exception** | An occurrence that has been modified (different time, title, etc.) or cancelled. Represented as a VEVENT with a RECURRENCE-ID embedded inline in the master VCALENDAR. May also appear as a separate database row for legacy or server-returned standalone exception resources. |
| **EXDATE** | A date added to the master event to exclude a specific occurrence from the RRULE expansion. Used **only** by the delete-occurrence flow; exception edits use RECURRENCE-ID instead (RFC 5545-compliant parsers substitute the exception VEVENT automatically). |
| **RECURRENCE-ID** | A property on the exception VEVENT that identifies which occurrence it replaces (the original, unmodified start time). |
| **UID** | A globally unique identifier shared by the master and **all its exceptions**. Required by RFC 5545. |

### Key Constraint

Exceptions **must** share the same UID as their master event. This is what links them to the series. The `RECURRENCE-ID` value (not the UID) differentiates an exception from its master.

## 2. Database Storage

Events are stored in the `Event` table (SQLite, managed by C++ sync engine) with these indexed columns:

| Column | Description |
|---|---|
| `id` | Stable UUID: `idForEvent(accountId, calendarId, uid, recurrenceId)` |
| `icsuid` | The ICS UID value (same for master and all exceptions) |
| `recurrenceId` | Empty string for master; original occurrence datetime for exceptions (e.g., `"20240116T100000Z"`) |
| `ics` | Full ICS text (VCALENDAR + VEVENT) |
| `recurrenceStart` | Unix seconds: start of first occurrence (master) or exception's new start |
| `recurrenceEnd` | Unix seconds: end of last occurrence (master) or exception's new end |
| `etag` | CalDAV ETag for concurrency control |
| `href` | CalDAV resource path (e.g., `/calendars/user/calendar/uid.ics`) |

### Master vs. Exception Identification

```typescript
// In TypeScript (Event model)
event.isRecurrenceException()  // returns true if recurrenceId != ""

// In C++ (Event model)
event->isRecurrenceException()  // same
```

## 3. Calendar Data Source: How Events Are Displayed

**File:** `app/internal_packages/main-calendar/lib/core/calendar-data-source.ts`

### Query

The `CalendarDataSource` queries all events (masters and exceptions) whose `recurrenceStart`/`recurrenceEnd` overlaps the visible date range. This returns raw `Event` rows.

### Grouping by UID

Events are grouped by `icsuid` to associate masters with their exceptions:

```typescript
const eventsByUid = new Map<string, Event[]>();
for (const event of results) {
  eventsByUid.get(event.icsuid).push(event);
}
```

### Expansion via ical-expander

For each UID group:

1. **Master present**: Pass the master's ICS to `ical-expander`. It uses the RRULE and EXDATE to generate all `EventOccurrence` objects for the visible range. If the master ICS contains inline exceptions (separate VEVENTs with RECURRENCE-ID), ical-expander substitutes them for their original occurrences automatically.

2. **Standalone exception records**: Any exception database row whose `recurrenceStart` was NOT already covered by master expansion is added as a standalone `EventOccurrence`. This handles servers that store exceptions as separate CalDAV resources.

3. **No master present**: Only standalone exceptions are shown.

### EventOccurrence Object

The UI never works with raw `Event` models directly; it works with `EventOccurrence` objects:

```typescript
interface EventOccurrence {
  id: string;       // Format: "${eventDbId}-e${idx}"
  start: number;    // Unix seconds (occurrence-specific, NOT master's start)
  end: number;      // Unix seconds
  isRecurring: boolean;
  isException: boolean;
  // ... title, location, attendees, etc.
}
```

The `id` format (`${masterEventId}-e${idx}`) is important: `parseEventIdFromOccurrence()` strips the `-e${idx}` suffix to get the underlying database `Event.id`.

## 4. Event Editing: "Edit This Occurrence" vs. "Edit All"

**File:** `app/internal_packages/main-calendar/lib/core/calendar-event-popover.tsx`

When the user saves changes to a recurring event (not already an exception), a dialog appears:

```typescript
const choice = await showRecurringEventDialog('edit', event.title);
// Returns 'this-occurrence' | 'all-occurrences' | 'cancel'
```

### "All Occurrences" Path (`_saveAllOccurrences`)

1. Apply property edits (title, location, times, etc.) to the master's ICS.
2. Update the RRULE if the repeat option changed.
3. Queue `SyncbackEventTask.forUpdating(masterEvent)`.

The master event is PUTted to its existing CalDAV href. All occurrences change.

### "This Occurrence Only" Path (`_saveOccurrenceException`)

Per RFC 4791 §4.1, exception VEVENTs with the same UID must share the same CalDAV resource as their master. Mailspring embeds the exception inline in the master VCALENDAR:

1. Call `ICSEventHelpers.createRecurrenceException(masterIcs, originalOccurrenceStart, newStart, newEnd, isAllDay)`.
2. This returns:
   - `masterIcs`: updated master VCALENDAR containing **both** the master VEVENT and the new exception VEVENT (no separate resource; no EXDATE added)
   - `recurrenceId`: the RECURRENCE-ID value (e.g., `"20240116T100000Z"`)
3. Call `ICSEventHelpers.applyEditsToException(masterIcs, recurrenceId, { summary, location, description, attendees })` to apply property edits to the inline exception VEVENT.
4. Update `masterEvent.ics` to the result.
5. Queue **one task**: `SyncbackEventTask.forUpdating(masterEvent)`.

The master's CalDAV resource is PUTted once; the server receives the full VCALENDAR containing both VEVENTs. ical-expander automatically handles inline exceptions.

**Re-editing an already-excepted occurrence:** `createRecurrenceException()` has upsert semantics — it finds and replaces the existing exception VEVENT in the master VCALENDAR rather than creating a duplicate.

### Drag-and-Drop Path

When an event is dragged/resized, `modifyEventWithRecurringSupport()` in `recurring-event-actions.ts` shows the same dialog and routes to either `modifyAllOccurrences()` or `createOccurrenceException()`. The logic is identical (single task, inline exception).

## 5. ICS Generation for Exceptions

**File:** `app/src/ics-event-helpers.ts` — `createRecurrenceException()` and `applyEditsToException()`

### `createRecurrenceException()` — What It Does

1. Parses master VCALENDAR with ical.js.
2. Checks for an existing exception VEVENT with the same RECURRENCE-ID (upsert: removes it if found).
3. Clones the master VEVENT to create the exception VEVENT:
   - Removes `RRULE`, `RDATE`, `EXDATE` (exception is a single instance)
   - Adds `RECURRENCE-ID` pointing to the original occurrence start
   - Updates `DTSTART`/`DTEND` to the new times
   - Increments `SEQUENCE`; updates `DTSTAMP`
4. Adds the exception VEVENT directly to the existing master VCALENDAR (`masterRoot.addSubcomponent(exceptionVevent)`).
5. Returns `{ masterIcs, recurrenceId }` — the updated master VCALENDAR now contains both VEVENTs.

**Note:** No `EXDATE` is added to the master VEVENT. EXDATE is only used by the delete-occurrence flow (`addExclusionDate`). RFC 5545-compliant parsers (including ical-expander) automatically substitute the exception VEVENT for its corresponding occurrence when a RECURRENCE-ID match is found.

### `applyEditsToException()` — What It Does

Finds the exception VEVENT in a master VCALENDAR by RECURRENCE-ID and applies property edits (summary, location, description, attendees). This is necessary because `updateEventProperty` and `updateAttendees` target the first VEVENT (the master), not a specific exception.

### Timezone Handling

The original timezone from the master's DTSTART is preserved for the exception's DTSTART/DTEND and RECURRENCE-ID.

### RECURRENCE-ID Format

For all-day events: `YYYYMMDD` (date only, local time)
For timed events: `YYYYMMDDTHHMMSSz` (UTC, e.g., `20240116T100000Z`)

## 6. CalDAV Synchronization

### Writing (C++: `DAVWorker::writeAndResyncEvent`)

When a `SyncbackEventTask` is processed remotely:

1. Look up the event in the local DB.
2. Determine the CalDAV resource href:
   - If `event->href()` is stored (previously synced): use it.
   - Otherwise, generate from UID: `{calendarPath}{uid}.ics`

   Both master events and inline exceptions share the same `{uid}.ics` resource (RFC 4791 §4.1). The master ICS now contains all exception VEVENTs inline, so a single PUT delivers the complete state.

3. PUT the event ICS (VCALENDAR with master VEVENT + any inline exception VEVENTs) to the resolved URL.
4. Multiget the href to get the server's canonical version (new etag, canonical href).
5. Update the local event record with server data.

### Reading (C++: CalDAV sync)

When syncing from the server, each `.ics` resource may contain multiple VEVENTs (master + inline exceptions). The sync engine processes each VEVENT separately:

```cpp
for (auto icsEvent : cal.Events) {
    // Query: WHERE icsuid = ? AND recurrenceId = ?
    auto existing = store->find<Event>(query);
    if (existing) {
        existing->applyICSEventData(etag, href, icsData, icsEvent);
        store->save(existing.get());
    } else {
        // Create new record (stable ID from UID + recurrenceId)
    }
}
```

This handles both "inline exception" storage (both VEVENTs in one resource) and "separate resource" storage (exception at its own href). The key is the lookup by `icsuid + recurrenceId`, not by href.

### Deletion Cascade

When a master event's CalDAV resource is deleted, the sync engine also deletes all local exception records sharing the same `icsuid`. Exceptions at separate hrefs are deleted independently.

## 7. Undo/Redo for Event Edits

**File:** `app/src/flux/tasks/syncback-event-task.ts`

### Pattern

Uses the "snapshot" pattern from `docs/undo-redo-task-pattern.md`:

```typescript
const undoData = { ics: event.ics, recurrenceStart: ..., recurrenceEnd: ... };
event.ics = newIcs;
Actions.queueTask(SyncbackEventTask.forUpdating({ event, undoData, description: 'Edit event' }));
```

`createUndoTask()` swaps `newData` and `undoData` to reverse the change.

### Exception Undo

Because "edit this occurrence" is now a **single master update task**, undo reverses the entire master ICS change in one step. The inline exception VEVENT is removed and the original occurrence is fully restored — no orphaned records, no separate undo needed for exception creation.

## 8. Data Flow Summary

```
User edits occurrence → "Edit this only" dialog

TypeScript:
  createRecurrenceException(masterIcs)
    → masterIcs (master VCALENDAR with inline exception VEVENT)
  applyEditsToException(masterIcs, recurrenceId, { title, location, ... })
    → updatedMasterIcs

  masterEvent.ics = updatedMasterIcs
  Actions.queueTask(SyncbackEventTask.forUpdating(master))  // single task!

C++ TaskProcessor:
  performLocalSyncbackEvent(task)  // write updated master to local DB immediately
  performRemoteSyncbackEvent(task) // PUT to CalDAV server

C++ DAVWorker::writeAndResyncEvent(master):
  PUT → /calendar/{uid}.ics  (master VCALENDAR with inline exception VEVENT)
  GET → update local master record with new etag + full ICS

TypeScript CalendarDataSource:
  Query by date range → gets master record (ICS contains inline exception)
  Group by icsuid
  ical-expander on master.ics → expands RRULE, substitutes exception VEVENT for its occurrence
  Standalone exception fallback → handles legacy separate-resource exception DB records
  → EventOccurrence[] rendered in calendar view
```
